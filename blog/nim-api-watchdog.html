Building an API Watchdog That Actually Auto-Resumes

June 19, 2026 · 5 min read
Python Automation Hermes Agent NVIDIA NIM

Here's the scenario: you're running an AI agent on a free-tier API, deep into a multi-step task, and BAM — HTTP 429, rate limited. Your agent dies, you check back an hour later, the API is fine, but now you have to tell it "continue" and re-explain what you were doing. Every. Single. Time.

I got tired of this loop. So I built a watchdog that pings the API, detects when it recovers, and auto-resumes the task without any human input. Here's how it works.

The Problem with "Notify Me" Approaches

Most watchdog patterns work like this:

[Cron Job] --> ping API --> API down? --> send "still down" msg API up? --> send "API is back!" msg

That's fine for server monitoring. But for an AI agent, "API is back!" just means you still have to manually say "okay, continue what you were doing." The agent doesn't remember what it was working on.

The fix: make the watchdog an agent itself. When it detects recovery, it reads a task description file and resumes the work autonomously.

The Architecture

[Agent working on task] | v (writes task description to file before timeout) [/tmp/nim_active_task.txt: "Setting up nginx reverse proxy"] | v [API rate limited - agent dies] | v [Watchdog cron - every 1 hour] |-- ping API (1 token) |-- Still down? --> silent, try again next hour |-- Back UP + no task? --> notify user "API is back!" |-- Back UP + HAS task? --> spawn agent to auto-resume | v [New agent reads task, picks up where we left off]

Key Design Decisions

Silence When Down, Speak When Up

The most important rule: no output = no message. When the API is still down, the script exits with code 1 and produces no stdout. The cron system sees no output and sends nothing. Your phone doesn't buzz every hour with "still down!"

When the API recovers, the script prints a single signal line to stdout. The cron system sees output and triggers the agent.

State Tracking, Not Poll Spam

A JSON state file tracks the last known status (up/down/unknown). The script only produces output on a status change (down to up). If the API was already up and is still up? Silent. No spam.

Task Bookmarking

Before the agent times out (or proactively), it writes a brief description of what it's working on to a task file. The watchdog reads this file and includes it in the recovery signal. The spawned agent gets the context it needs to continue.

The Signal Protocol

On recovery, the script prints exactly one line:

NIM_API_RECOVERED|NO_TASK
NIM_API_RECOVERED|HAS_TASK|Setting up nginx reverse proxy

The cron system parses this and routes accordingly:

  • NO_TASK → just tell the user the API is back
  • HAS_TASK|description → spawn an agent with the task description as context

The Script

The full Python script is available on GitHub:

github.com/cappy-dev/nim-api-watchdog

Key features:

  • Pure Python 3, no pip dependencies (uses urllib)
  • 1 token per health check — negligible cost
  • Reads API key from environment variable or .env file
  • Model configurable via NIM_MODEL env var
  • 15-second timeout per check
  • State file at /tmp/nim_watchdog_state.json
  • Task file at /tmp/nim_active_task.txt

Setting It Up with Hermes Agent

hermes cron add \
  --name "NIM Resume Watchdog" \
  --schedule "every 1h" \
  --script nim_watchdog.py \
  --prompt "The NIM API just recovered. Parse the signal. If HAS_TASK, auto-resume without asking. If NO_TASK, notify user." \
  --deliver origin

Token Cost Analysis

NIM's free tier: 1,000 credits/month. 1 credit = 1,000 tokens.

  • Each health check: 1 completion token
  • Every hour: 24 checks/day = 24 tokens/day
  • Monthly: ~720 tokens = 0.72 credits
  • That's 0.072% of your monthly budget

Basically free. The watchdog costs less than a single real conversation turn.

Honest Limitations

This isn't a perfect context restoration system. The resumed agent won't have the full conversation history — it only gets the task description as a starting point. Think of it as a bookmark, not a save state.

For tasks that need precise context (debugging a specific error message, editing a specific file at a specific line), the task description might not be enough. For broader tasks ("set up DNS for my homelab" or "write a blog post about X"), it works great.

What About Fallback Providers?

The other approach: don't wait at all. Configure fallback providers so your agent automatically switches to a different API when NIM is down. Hermes Agent supports three layers of resilience:

  • Credential pools — rotate between multiple API keys
  • Model fallback — switch to a different model/provider automatically
  • Auxiliary task fallback — use a lighter model for specific subtasks

The watchdog and fallback approaches complement each other. Fallbacks keep the agent running right now. The watchdog recovers it when everything is down and comes back later.
